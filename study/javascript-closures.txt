## What a Closure Actually Is

A closure is a function bundled with its **lexical environment** — the scope chain at the point where the function was *defined*, not where it's called.

```js
function outer() {
  const secret = 42  // Lives in outer's scope

  function inner() {
    console.log(secret)  // inner "closes over" secret
  }

  return inner  // Return the function, scope travels with it
}

const fn = outer()  // outer() has returned, its stack frame is gone
fn()  // 42 — but secret is still alive, held by the closure
```

The garbage collector cannot collect `secret` because `inner` holds a live reference to it through its closure. The scope lives as long as the function lives.

## Factory Functions — Closures for Encapsulation

The classic use case: create private state that only your returned functions can access.

```js
function makeCounter(start = 0) {
  let count = start  // Private — no external access

  return {
    increment() { count++ },
    decrement() { count-- },
    reset()     { count = start },
    value()     { return count },
  }
}

const counter = makeCounter(10)
counter.increment()
counter.increment()
console.log(counter.value())  // 12
console.log(counter.count)    // undefined — truly private
```

Each call to `makeCounter` creates a **new closure** with its own `count`:

```js
const a = makeCounter()
const b = makeCounter()
a.increment()
a.increment()
console.log(a.value())  // 2
console.log(b.value())  // 0 — independent state
```

## Memoization — Closures for Caching

```js
function memoize(fn) {
  const cache = new Map()  // Closed over by the returned function

  return function(...args) {
    const key = JSON.stringify(args)
    if (cache.has(key)) {
      console.log('cache hit')
      return cache.get(key)
    }
    const result = fn(...args)
    cache.set(key, result)
    return result
  }
}

const expensiveCalc = memoize((n) => {
  // Simulate expensive computation
  return n * n
})

expensiveCalc(4)  // Computed: 16
expensiveCalc(4)  // cache hit: 16
expensiveCalc(5)  // Computed: 25
```

## The Closure-in-Loop Gotcha

This is the most common closure bug in JavaScript. Every interviewer asks about it.

```js
// ❌ Classic bug
for (var i = 0; i < 3; i++) {
  setTimeout(() => console.log(i), 100)
}
// Prints: 3, 3, 3
// All three callbacks close over the *same* i (var is function-scoped)
// By the time they run, the loop has ended and i === 3
```

**Fix 1: Use `let` (block-scoped — creates new binding per iteration)**
```js
for (let i = 0; i < 3; i++) {
  setTimeout(() => console.log(i), 100)
}
// Prints: 0, 1, 2  ✓
```

**Fix 2: IIFE to capture the value**
```js
for (var i = 0; i < 3; i++) {
  ;(function(j) {
    setTimeout(() => console.log(j), 100)
  })(i)
}
// Prints: 0, 1, 2  ✓ — j is a new variable per IIFE call
```

**Fix 3: Extract to a factory function**
```js
function makeLogger(n) {
  return () => console.log(n)  // n is captured at call time
}

for (var i = 0; i < 3; i++) {
  setTimeout(makeLogger(i), 100)
}
// Prints: 0, 1, 2  ✓
```

## Module Pattern — Closures for Information Hiding

Before ES modules, closures were the primary way to create modules with private state:

```js
const UserStore = (function() {
  // Private — inaccessible outside this IIFE
  let users = []
  let nextId = 1

  function validate(user) {
    return user.name && user.email
  }

  // Public API
  return {
    add(user) {
      if (!validate(user)) throw new Error('Invalid user')
      users.push({ ...user, id: nextId++ })
    },
    getAll() {
      return [...users]  // Return copy, not reference
    },
    findById(id) {
      return users.find(u => u.id === id)
    },
    count() {
      return users.length
    }
  }
})()

UserStore.add({ name: 'Allen', email: 'a@example.com' })
console.log(UserStore.count())  // 1
console.log(UserStore.users)    // undefined — private
```

## Partial Application and Currying

Closures enable partial application — fixing some arguments and returning a function for the rest:

```js
function multiply(a) {
  return function(b) {  // Closes over a
    return a * b
  }
}

const double = multiply(2)
const triple = multiply(3)

double(5)  // 10
triple(5)  // 15

// Arrow function shorthand
const add = a => b => a + b

const add10 = add(10)
add10(5)   // 15
add10(20)  // 30
```

Real-world example — a generic API caller with a base URL pre-configured:

```js
function createApiClient(baseURL) {
  async function request(method, path, body) {
    const res = await fetch(`${baseURL}${path}`, {
      method,
      headers: { 'Content-Type': 'application/json' },
      body: body ? JSON.stringify(body) : undefined,
    })
    if (!res.ok) throw new Error(`${method} ${path} → ${res.status}`)
    return res.json()
  }

  return {
    get:    (path) => request('GET', path),
    post:   (path, body) => request('POST', path, body),
    put:    (path, body) => request('PUT', path, body),
    delete: (path) => request('DELETE', path),
  }
}

const api = createApiClient('https://api.example.com')
const users = await api.get('/users')
await api.post('/users', { name: 'Allen' })
```

## Closures in React Hooks

React hooks rely entirely on closures. Each render creates new closures, which is why the rules of hooks (no conditionals, no loops) exist — React tracks hooks by call order, and the closures must be stable.

```js
function useDebounce(value, delay) {
  const [debouncedValue, setDebouncedValue] = useState(value)

  useEffect(() => {
    // This closure captures 'value' at this render's snapshot
    const timer = setTimeout(() => {
      setDebouncedValue(value)  // Always the 'value' from *this* effect's render
    }, delay)

    return () => clearTimeout(timer)  // Cleanup closes over 'timer'
  }, [value, delay])

  return debouncedValue
}
```

## Memory Considerations

Closures can leak memory if they inadvertently hold large objects:

```js
function processLargeData() {
  const bigArray = new Array(1_000_000).fill(0)  // 8 MB

  // ❌ This closure holds bigArray alive indefinitely
  return () => console.log(bigArray.length)
}

// ✅ Only keep what you need
function processLargeData() {
  const bigArray = new Array(1_000_000).fill(0)
  const length = bigArray.length  // Extract what's needed
  // bigArray can now be GC'd after the function returns

  return () => console.log(length)
}
```

Watch for closures inside event listeners that aren't cleaned up, and closures that reference DOM nodes holding large subtrees.

## Mental Model Summary

1. **A closure = function + its birth scope**
2. The captured scope lives as long as the closure lives
3. Multiple closures over the same scope share the same variables (gotcha in loops with `var`)
4. `let`/`const` in loops creates per-iteration bindings — no sharing
5. Factory functions + closures = the OOP pattern without classes
