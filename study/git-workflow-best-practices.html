Git Workflow Best Practices

Conventional commits, branch naming, rebase vs merge, and the exact workflow that keeps a solo dev project history clean.

Git Workflow Best Practices

Conventional Commits — Structured History

Format: <type>(<scope>): <description>

feat(auth): add OAuth2 login with Google
fix(nav): correct mobile menu z-index stacking
docs(api): update endpoint documentation for v2
refactor(db): extract connection pooling to separate module
test(payment): add integration tests for Stripe webhook
chore(deps): update Next.js from 14.1 to 14.2
perf(images): lazy-load below-fold portfolio images
ci: add Lighthouse performance budget check

Types:

TypeWhen
featNew feature
fixBug fix
refactorRestructure without behavior change
perfPerformance improvement
testAdd or fix tests
docsDocumentation only
styleFormatting (no logic change)
choreBuild process, dependencies
ciCI/CD pipeline

A breaking change adds ! after the type: feat!: remove deprecated v1 API endpoints

Branch Naming

feature/  — new functionality
fix/      — bug fixes
refactor/ — restructuring
chore/    — maintenance
release/  — release prep

Examples:
  feature/study-page-search
  fix/mobile-nav-overflow
  refactor/extract-api-client
  chore/update-tailwind-v4
  release/v2.1.0

Keep branch names lowercase, hyphenated, and short enough to read at a glance.

The Daily Workflow

# 1. Start from latest main
git switch main
git pull --ff-only  # Fail loudly if not a fast-forward

# 2. Create feature branch
git switch -c feature/study-search

# 3. Work — commit early and often
git add src/components/SearchBar.jsx
git commit -m "feat(study): add search bar component"

git add src/hooks/useSearch.js
git commit -m "feat(study): implement useSearch with debounce"

git add src/app/study/page.jsx
git commit -m "feat(study): integrate search into study page"

# 4. Keep up with main
git fetch origin
git rebase origin/main  # Reapply your commits on top of latest main

# 5. Clean up commits before PR (squash noise)
git rebase -i origin/main

# 6. Push and open PR
git push -u origin feature/study-search

Rebase vs Merge

Merge (preserves history, adds merge commit):

  main:    A──B──C──────────M
                  \        /
  feature:         D──E──F

Result: "Merge feature/xyz into main" commit
Use for: merging release branches, preserving context

Rebase (linear history, replays commits):

  main:    A──B──C──D'──E'──F'
  feature: (gone — rebased onto main)

Result: Clean linear history, as if you always had the latest main
Use for: feature branches before merging to main

The golden rule: Never rebase shared branches (main, release). Only rebase your own local branches.

Interactive Rebase — Cleaning Commits

# Rebase last 4 commits interactively
git rebase -i HEAD~4

# Editor opens with:
pick abc1234 WIP save progress
pick def5678 fix typo
pick ghi9012 more WIP
pick jkl3456 feat: complete search feature

# Change to:
squash abc1234 WIP save progress    ← squash into next
squash def5678 fix typo             ← squash into next
squash ghi9012 more WIP             ← squash into next
reword jkl3456 feat: complete search feature  ← keep, edit message

Commands: pick (keep), squash/s (merge into previous), fixup/f (squash, discard message), reword/r (edit message), drop/d (delete commit)

Finding Bugs with Git Bisect

When you know a bug was introduced "sometime in the last 100 commits":

git bisect start
git bisect bad               # Current commit is broken
git bisect good v2.0.0       # This tag was known good

# Git checks out the midpoint commit
# Test: does the bug exist?

git bisect bad   # Bug exists in this commit — search left half
# or
git bisect good  # Bug doesn't exist — search right half

# Repeat until Git identifies the first bad commit
# Git tells you: "abc1234 is the first bad commit"

git bisect reset  # Return to original HEAD

Automate with a test script:

git bisect run npm test -- --grep "search returns results"
# Git automatically runs the test on each midpoint commit

Recovering Lost Work with Reflog

git reflog shows every position HEAD has been — commits, checkpoints, rebases. It's your safety net.

# You accidentally reset --hard and lost commits
git reflog
# 4a1b2c3 HEAD@{0}: reset: moving to HEAD~3  ← disaster
# 7d8e9f0 HEAD@{1}: commit: feat: complete search  ← your work
# ...

# Recover:
git reset --hard 7d8e9f0  # Or: git checkout -b recovery 7d8e9f0

Reflog entries expire after 90 days by default — always check early after an accident.

Commit Hygiene Rules

One logical change per commit. A commit should answer: "what changed and why?" in a single sentence.

# ❌ Too broad
git commit -m "fix stuff and update deps and add tests"

# ✅ Atomic commits
git commit -m "fix(nav): correct active link detection on nested routes"
git commit -m "chore(deps): upgrade framer-motion to 11.0"
git commit -m "test(nav): add test for nested route active state"

Write the why, not the what. The diff shows what changed — the message explains why.

# ❌ Describes the what (already obvious from diff)
git commit -m "change button color to blue"

# ✅ Explains the why
git commit -m "style(button): use brand blue per updated design tokens

Previous #333 color failed WCAG AA contrast on dark backgrounds.
Updated to match the design system token --color-brand-700."

Git Aliases for Speed

# ~/.gitconfig
[alias]
  st   = status -sb
  lg   = log --oneline --graph --decorate --all
  co   = checkout
  sw   = switch
  br   = branch --sort=-committerdate
  undo = reset --soft HEAD~1   # Undo last commit, keep changes staged
  wip  = !git add -A && git commit -m "chore: WIP checkpoint"
  sync = !git fetch origin && git rebase origin/main

# Usage
git lg       # Visual branch graph
git undo     # Un-commit last change, keep it staged
git wip      # Quick checkpoint commit
git sync     # Update feature branch from main

Pre-commit Hooks with Husky

Automate quality gates before every commit:

npm install --save-dev husky lint-staged
npx husky init

// package.json
{
  "lint-staged": {
    "*.{js,jsx,ts,tsx}": ["eslint --fix", "prettier --write"],
    "*.{css,md}": "prettier --write"
  }
}

# .husky/pre-commit
npx lint-staged

Now every commit automatically lints and formats changed files — no style debates in PR reviews.
