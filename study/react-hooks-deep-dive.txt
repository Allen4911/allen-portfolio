## The Core Mental Model: Closures Over Snapshots

Every React render creates a **snapshot** of props and state. Hooks are closures captured at that snapshot. This one idea explains most hook bugs.

```js
function Counter() {
  const [count, setCount] = useState(0)

  // This setTimeout closes over the *snapshot* of count at render time
  function handleAlertClick() {
    setTimeout(() => {
      alert('You clicked: ' + count)  // Always the count at click time
    }, 3000)
  }

  return <button onClick={handleAlertClick}>Click me</button>
}
```

If you click when count is 0, then increment to 5, the alert still shows 0. That's not a bug — it's the snapshot model working correctly.

## useState — Beyond the Basics

### Functional Updates

When new state depends on previous state, always use the functional form:

```js
// ❌ Stale closure risk
setCount(count + 1)

// ✅ Always reads the latest state
setCount(prev => prev + 1)
```

This matters inside `useCallback`, `useEffect`, and async handlers where the closure may be stale.

### Object State Patterns

React does **not** shallow-merge object state (unlike class `setState`). You must spread manually:

```js
const [user, setUser] = useState({ name: 'Allen', age: 30 })

// ❌ Replaces entire state — loses other fields
setUser({ age: 31 })

// ✅ Spread to preserve other fields
setUser(prev => ({ ...prev, age: 31 }))
```

### Lazy Initialization

For expensive initial computations, pass a function (not a value):

```js
// ❌ parseSavedData runs on every render, result discarded after first
const [data, setData] = useState(parseSavedData(rawJSON))

// ✅ Function runs only once on mount
const [data, setData] = useState(() => parseSavedData(rawJSON))
```

## useEffect — The Synchronization Primitive

`useEffect` is not a lifecycle hook. It's a **synchronization mechanism** — it keeps external systems (DOM, network, timers) in sync with React state.

```js
useEffect(() => {
  // Effect: setup / subscribe
  const sub = websocket.subscribe(userId, onMessage)

  // Cleanup: runs before next effect and on unmount
  return () => sub.unsubscribe()
}, [userId, onMessage]) // Re-run when these change
```

### Dependency Array Rules

| Dependency | Behavior |
|-----------|----------|
| Omitted | Runs after every render |
| `[]` | Runs once on mount |
| `[a, b]` | Runs when a or b changes |

The ESLint rule `exhaustive-deps` enforces that all variables used inside the effect appear in the array. **Follow it.** Suppressing it with `// eslint-disable` almost always introduces stale closure bugs.

### Common Effect Patterns

**Data fetching with cleanup:**
```js
useEffect(() => {
  let cancelled = false

  async function fetchUser() {
    const data = await getUser(userId)
    if (!cancelled) setUser(data)  // Guard against race conditions
  }

  fetchUser()
  return () => { cancelled = true }
}, [userId])
```

**Event listener:**
```js
useEffect(() => {
  function handleResize() {
    setWidth(window.innerWidth)
  }
  window.addEventListener('resize', handleResize)
  return () => window.removeEventListener('resize', handleResize)
}, [])
```

## useRef — Two Distinct Uses

### 1. Accessing DOM Nodes

```jsx
function AutoFocusInput() {
  const inputRef = useRef(null)

  useEffect(() => {
    inputRef.current.focus()
  }, [])

  return <input ref={inputRef} />
}
```

### 2. Mutable Instance Variable

A ref is the only way to hold a mutable value that **doesn't trigger re-renders**:

```js
function Timer() {
  const [count, setCount] = useState(0)
  const intervalRef = useRef(null)

  function start() {
    intervalRef.current = setInterval(() => {
      setCount(c => c + 1)
    }, 1000)
  }

  function stop() {
    clearInterval(intervalRef.current)
  }

  return <><button onClick={start}>Start</button><button onClick={stop}>Stop</button></>
}
```

Use refs for: timer IDs, previous values, scroll positions, abort controllers — anything that needs to persist across renders without causing re-renders.

## useCallback and useMemo — When to Memoize

Both hooks trade memory for computation. They're **not free** — use them only when you have a measured performance problem.

### useCallback — Stable Function References

```js
// ❌ New function reference on every render → child always re-renders
const handleSubmit = () => { submitForm(data) }

// ✅ Same reference unless 'data' changes
const handleSubmit = useCallback(() => {
  submitForm(data)
}, [data])
```

**When it matters:** Only useful when the function is passed to a memoized child (`React.memo`) or appears in another hook's dependency array.

### useMemo — Expensive Computations

```js
// ❌ filteredItems recomputed on every render
const filteredItems = items.filter(i => i.active && i.category === cat)

// ✅ Only recomputed when items or cat changes
const filteredItems = useMemo(
  () => items.filter(i => i.active && i.category === cat),
  [items, cat]
)
```

**Rule of thumb:** Profile first. Memoize only if:
- The computation is measurably slow (>1ms)
- The result is used in a dependency array of another hook
- The result is passed as props to `React.memo` components

## useReducer — State Machines for Complex State

When state transitions are complex or involve multiple sub-values, `useReducer` beats `useState`:

```js
const initialState = { count: 0, loading: false, error: null }

function reducer(state, action) {
  switch (action.type) {
    case 'INCREMENT': return { ...state, count: state.count + 1 }
    case 'FETCH_START': return { ...state, loading: true, error: null }
    case 'FETCH_SUCCESS': return { ...state, loading: false, count: action.payload }
    case 'FETCH_ERROR': return { ...state, loading: false, error: action.error }
    default: return state
  }
}

function Counter() {
  const [state, dispatch] = useReducer(reducer, initialState)

  async function fetchCount() {
    dispatch({ type: 'FETCH_START' })
    try {
      const count = await api.getCount()
      dispatch({ type: 'FETCH_SUCCESS', payload: count })
    } catch (e) {
      dispatch({ type: 'FETCH_ERROR', error: e.message })
    }
  }

  return (
    <div>
      {state.loading && <Spinner />}
      {state.error && <Error message={state.error} />}
      <p>Count: {state.count}</p>
      <button onClick={() => dispatch({ type: 'INCREMENT' })}>+</button>
    </div>
  )
}
```

## Custom Hooks — Extracting Stateful Logic

Custom hooks are the primary reuse mechanism in React. Any function that starts with `use` and calls built-in hooks is a custom hook.

```js
// useFetch — generic data fetching with loading/error states
function useFetch(url) {
  const [state, dispatch] = useReducer(
    (s, a) => ({ ...s, ...a }),
    { data: null, loading: true, error: null }
  )

  useEffect(() => {
    let cancelled = false
    dispatch({ loading: true, error: null })

    fetch(url)
      .then(r => r.json())
      .then(data => { if (!cancelled) dispatch({ data, loading: false }) })
      .catch(err => { if (!cancelled) dispatch({ error: err.message, loading: false }) })

    return () => { cancelled = true }
  }, [url])

  return state
}

// Usage
function UserProfile({ userId }) {
  const { data, loading, error } = useFetch(`/api/users/${userId}`)

  if (loading) return <Spinner />
  if (error) return <Error message={error} />
  return <Profile user={data} />
}
```

## Common Hook Mistakes

| Mistake | Fix |
|---------|-----|
| `setCount(count + 1)` in async handler | Use `setCount(c => c + 1)` |
| Missing dependency in `useEffect` | Add it or extract to a ref |
| Calling a hook inside a condition | Move condition inside the hook |
| Creating objects/arrays in JSX deps | Memoize with `useMemo` or move outside component |
| Using `useLayoutEffect` when `useEffect` works | `useLayoutEffect` blocks paint — only for DOM measurements |

---

## Hook Decision Guide

```
Need reactive value?
  └─ One value → useState
  └─ Multiple related values + complex transitions → useReducer

Need a side effect?
  └─ After render, okay to be async → useEffect
  └─ Before paint, DOM measurement → useLayoutEffect

Need DOM access?
  └─ useRef

Need to persist value without re-render?
  └─ useRef

Need stable function reference?
  └─ useCallback (only if passed to memo'd child or in dep array)

Need to cache computation?
  └─ useMemo (only if provably expensive)

Repeating same stateful logic?
  └─ Extract to custom hook
