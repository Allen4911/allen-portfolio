Deploying an AI Agent to production is harder in terms of designing the evaluation framework than in model selection. This post summarizes failure patterns and measurement metrics encountered while operating real agent systems.

## What is an AI Agent?

An AI Agent is different from a simple LLM call. It uses tools, performs multi-step reasoning, and acts autonomously to achieve goals.

```
User Goal
    ↓
Planning
    ↓
Tool Selection → Execution → Observation
    ↓
Goal Achievement Check
    ↓ (repeat if not achieved)
Final Response
```

## Key Evaluation Metrics

### 1. Task Completion Rate (TCR)

The most fundamental metric. The ratio of tasks the agent completes end-to-end.

```python
tcr = successful_completions / total_attempts

# Note: the definition of "completion" matters
# Bad definition: agent returns a response
# Good definition: the result the user wanted actually exists
```

TCR alone is insufficient. Agents frequently declare "complete" when the result is wrong (Hallucinated Completion).

### 2. Step Efficiency

Ratio of actual steps used versus minimum steps needed to achieve the goal.

```python
efficiency = optimal_steps / actual_steps
# 1.0 is ideal, lower means less efficient
```

This metric drops when the agent reads the same file 5 times or repeats already-failed approaches.

### 3. Tool Call Accuracy

The rate at which the agent calls the correct tool with the correct arguments.

```python
# Measured items
tool_selection_accuracy = correct_tool / total_tool_calls
argument_validity = valid_args / total_tool_calls
```

Common real-world failures: guessing wrong file paths, or using non-existent API parameters.

### 4. Hallucination Rate

The rate at which the agent asserts information it has not actually verified.

```python
# Detection method
def check_hallucination(claim, tool_outputs):
    # verify if claim can be derived from tool_outputs
    return not can_be_derived(claim, tool_outputs)
```

### 5. Recovery Rate

The rate at which the agent recovers on its own when an error occurs during a task. Good agents recognize failures and find alternative paths.

## Major Failure Modes

### Failure 1: Infinite Loop

The agent repeats the same task without being able to escape.

```
Attempt 1: Read file → Failed
Attempt 2: Read file → Failed
Attempt 3: Read file → Failed
... (infinite loop)
```

**Cause:** Not saving failure state to memory, or lack of ability to explore alternative paths.

**Countermeasure:** Set maximum retry count, add failure pattern detection logic.

```python
MAX_RETRIES = 3
retry_tracker = defaultdict(int)

def before_tool_call(tool, args):
    key = f"{tool}:{args}"
    retry_tracker[key] += 1
    if retry_tracker[key] > MAX_RETRIES:
        raise MaxRetriesExceeded(f"Stopping: {key} failed {MAX_RETRIES} times")
```

### Failure 2: Goal Drift

During a long task, the agent forgets the initial goal and drifts in a new direction.

**Example:** "Please modify file A" → Modifies files B, C, D, and A remains unchanged.

**Countermeasure:** Explicitly verify the connection to the original goal at each step.

```python
system_prompt = """
Current goal: {original_goal}

Before each action, ask yourself:
1. Does this action directly contribute to achieving the goal?
2. Am I doing work unrelated to the goal?
"""
```

### Failure 3: Premature Completion

Declares task complete even though the work is not actually done.

**Detection method:**

```python
def verify_completion(goal, agent_output):
    verifier_prompt = f"""
    Goal: {goal}
    Agent result: {agent_output}
    
    Was the goal fully achieved? List specifically any parts that were not achieved.
    """
    return call_llm(verifier_prompt)
```

The pattern of having a separate Verifier agent is effective.

### Failure 4: Context Overflow

In long conversations, early information is pushed out of the context window and forgotten.

**Symptoms:** Attempting already-completed work again, re-verifying previously confirmed information.

**Countermeasure:**

```python
# Store critical information in separate memory
memory = {
    "goal": original_goal,
    "completed_steps": [],
    "failed_attempts": [],
    "discovered_facts": []
}

# Update memory at each step
def update_memory(step_result):
    memory["completed_steps"].append(step_result)
```

### Failure 5: Tool Abuse

The agent over-relies on a single tool.

**Example:** Trying to find all information only through web search, without utilizing already-provided context.

## Building an Evaluation Environment

### Test Case Design

```python
test_cases = [
    {
        "task": "Find the formatDate function in src/utils/date.js and modify it to use ISO format",
        "setup": {"files": ["src/utils/date.js"]},
        "success_criteria": {
            "file_modified": "src/utils/date.js",
            "contains": "toISOString()",
            "tests_pass": True
        }
    }
]
```

### Automated Evaluation Pipeline

```python
async def evaluate_agent(agent, test_cases):
    results = []
    
    for tc in test_cases:
        setup_environment(tc["setup"])
        
        start_time = time.time()
        output = await agent.run(tc["task"])
        duration = time.time() - start_time
        
        success = verify_success(output, tc["success_criteria"])
        
        results.append({
            "task": tc["task"],
            "success": success,
            "duration": duration,
            "steps": output.step_count,
            "tool_calls": output.tool_calls
        })
    
    return compute_metrics(results)
```

## Production Monitoring

Metrics to track after deploying an agent:

| Metric | Threshold | Alert Condition |
|--------|-----------|-----------------|
| TCR | > 85% | Below 80% for 3 consecutive periods |
| Avg steps | &lt; 15 | Exceeds 20 |
| Avg response time | &lt; 30s | Exceeds 60s |
| Error rate | &lt; 5% | Exceeds 10% |
| Context usage | &lt; 70% | Exceeds 90% |

## Summary

The key to AI Agent evaluation is measuring "did the agent do the right thing," not "did the agent do something." Combining TCR, Step Efficiency, and Tool Call Accuracy allows early detection of most failure modes.

The most common mistake is selecting an agent based solely on LLM benchmark scores. Real-world performance can only be reliably measured through End-to-End evaluation in the actual task environment.
