## What is CLAUDE.md?

CLAUDE.md is a markdown file that Claude Code automatically loads as context at the start of every session. It defines how Claude should behave in a specific project or directory — enforcing conventions, team rules, and workflow patterns without manual re-prompting.

There are three scopes:
- `~/.claude/CLAUDE.md` — Global (applies to all projects)
- `/project-root/CLAUDE.md` — Project-level (checked into git)
- `/subdir/CLAUDE.md` — Directory-specific (for monorepos)

## Anatomy of a CLAUDE.md

```markdown
# Project: My App

## Tech Stack
- Next.js 15, TypeScript, Tailwind CSS
- Testing: Jest + React Testing Library

## Coding Standards
- Use TDD: write failing test before implementation
- No comments unless WHY is non-obvious
- Prefer editing existing files over creating new ones

## Git Conventions
- Commit message: type(scope): description
- Never push to main directly — use feature branches
```

Hooks are shell commands that run automatically on specific Claude Code events. Configured in `settings.json`:

## Why Standard ML Metrics Fall Short

Accuracy and F1 score were designed for classification tasks. AI agents — systems that reason, use tools, and produce long-form outputs — require a different evaluation framework. An agent can be correct in its final answer while being expensive, slow, or fragile.

## Core Evaluation Dimensions

### 1. Task Completion Rate

The fraction of tasks where the agent reaches a correct, verifiable final state.

```python
def task_completion_rate(results: list[dict]) -> float:
    completed = sum(1 for r in results if r['status'] == 'success' and r['correct'])
    return completed / len(results)
```

Measure separately for: simple tasks (single tool call), multi-step tasks (3+ tool calls), and recovery tasks (agent must handle a tool failure).

### 2. Token Efficiency

Tokens consumed per unit of useful work. An agent that uses 50k tokens to answer a 3-hop question is worse than one using 8k tokens.

```python
def token_efficiency(tasks: list[dict]) -> float:
    # Lower is better
    return sum(t['tokens_used'] for t in tasks) / sum(t['steps_correct'] for t in tasks)
```

Optimization levers:
- Prompt caching (Anthropic API: up to 90% cost reduction on repeated context)
- Context compaction (Claude Code's `/compact` command)
- RTK output filtering (60–90% reduction on shell command tokens)

### 3. Hallucination Rate

For agents that retrieve information, measure how often they assert facts not present in their tool outputs.

```python
def hallucination_rate(responses: list[dict]) -> float:
    hallucinated = sum(
        1 for r in responses
        if any(claim not in r['tool_outputs'] for claim in r['extracted_claims'])
    )
    return hallucinated / len(responses)
```

Mitigation: citations mode (Claude API), grounding assertions to retrieved documents.

### 4. Latency Distribution

P50/P95/P99 latency matters more than average. A slow tail means unreliable UX.

```python
import numpy as np

def latency_stats(latencies: list[float]) -> dict:
    return {
        'p50': np.percentile(latencies, 50),
        'p95': np.percentile(latencies, 95),
        'p99': np.percentile(latencies, 99),
        'mean': np.mean(latencies),
    }
```

## Multi-Agent Failure Modes

When multiple agents collaborate (PM → Developer → Reviewer), new failure categories emerge:

| Failure Mode | Description | Detection |
|-------------|-------------|-----------|
| Context drift | Agent A's output is misinterpreted by Agent B | Compare intent vs. execution |
| Role collision | Two agents perform the same task | Deduplication check on outputs |
| Infinite loop | Agent A waits for B, B waits for A | Timeout + circuit breaker |
| Cascade failure | One agent's error corrupts downstream agents | Checkpoint validation |

## The Core Mental Model: Closures Over Snapshots

Every React render creates a **snapshot** of props and state. Hooks are closures captured at that snapshot. This one idea explains most hook bugs.

```js
function Counter() {
  const [count, setCount] = useState(0)

  // This setTimeout closes over the *snapshot* of count at render time
  function handleAlertClick() {
    setTimeout(() => {
      alert('You clicked: ' + count)  // Always the count at click time
    }, 3000)
  }

  return <button onClick={handleAlertClick}>Click me</button>
}
```

If you click when count is 0, then increment to 5, the alert still shows 0. That's not a bug — it's the snapshot model working correctly.

## useState — Beyond the Basics

### Functional Updates

When new state depends on previous state, always use the functional form:

```js
// ❌ Stale closure risk
setCount(count + 1)

// ✅ Always reads the latest state
setCount(prev => prev + 1)
```

This matters inside `useCallback`, `useEffect`, and async handlers where the closure may be stale.

### Object State Patterns

React does **not** shallow-merge object state (unlike class `setState`). You must spread manually:

```js
const [user, setUser] = useState({ name: 'Allen', age: 30 })

// ❌ Replaces entire state — loses other fields
setUser({ age: 31 })

// ✅ Spread to preserve other fields
setUser(prev => ({ ...prev, age: 31 }))
```

### Lazy Initialization

For expensive initial computations, pass a function (not a value):

```js
// ❌ parseSavedData runs on every render, result discarded after first
const [data, setData] = useState(parseSavedData(rawJSON))

// ✅ Function runs only once on mount
const [data, setData] = useState(() => parseSavedData(rawJSON))
```

## useEffect — The Synchronization Primitive

`useEffect` is not a lifecycle hook. It's a **synchronization mechanism** — it keeps external systems (DOM, network, timers) in sync with React state.

```js
useEffect(() => {
  // Effect: setup / subscribe
  const sub = websocket.subscribe(userId, onMessage)

  // Cleanup: runs before next effect and on unmount
  return () => sub.unsubscribe()
}, [userId, onMessage]) // Re-run when these change
```

### Dependency Array Rules

| Dependency | Behavior |
|-----------|----------|
| Omitted | Runs after every render |
| `[]` | Runs once on mount |
| `[a, b]` | Runs when a or b changes |

The ESLint rule `exhaustive-deps` enforces that all variables used inside the effect appear in the array. **Follow it.** Suppressing it with `// eslint-disable` almost always introduces stale closure bugs.

### Common Effect Patterns

**Data fetching with cleanup:**
```js
useEffect(() => {
  let cancelled = false

  async function fetchUser() {
    const data = await getUser(userId)
    if (!cancelled) setUser(data)  // Guard against race conditions
  }

  fetchUser()
  return () => { cancelled = true }
}, [userId])
```

**Event listener:**
```js
useEffect(() => {
  function handleResize() {
    setWidth(window.innerWidth)
  }
  window.addEventListener('resize', handleResize)
  return () => window.removeEventListener('resize', handleResize)
}, [])
```

### 1. Accessing DOM Nodes

A ref is the only way to hold a mutable value that **doesn't trigger re-renders**:

```js
function Timer() {
  const [count, setCount] = useState(0)
  const intervalRef = useRef(null)

  function start() {

    clearInterval(intervalRef.current)

## useCallback and useMemo — When to Memoize

Both hooks trade memory for computation. They're **not free** — use them only when you have a measured performance problem.

### useCallback — Stable Function References

```js
// ❌ New function reference on every render → child always re-renders
const handleSubmit = () => { submitForm(data) }

// ✅ Same reference unless 'data' changes
const handleSubmit = useCallback(() => {
  submitForm(data)
}, [data])
```

**When it matters:** Only useful when the function is passed to a memoized child (`React.memo`) or appears in another hook's dependency array.

### useMemo — Expensive Computations

```js
// ❌ filteredItems recomputed on every render
const filteredItems = items.filter(i => i.active && i.category === cat)

// ✅ Only recomputed when items or cat changes
const filteredItems = useMemo(
  () => items.filter(i => i.active && i.category === cat),
  [items, cat]
)
```

**Rule of thumb:** Profile first. Memoize only if:
- The computation is measurably slow (>1ms)
- The result is used in a dependency array of another hook
- The result is passed as props to `React.memo` components

## useReducer — State Machines for Complex State

When state transitions are complex or involve multiple sub-values, `useReducer` beats `useState`:

```js
const initialState = { count: 0, loading: false, error: null }

function reducer(state, action) {
  switch (action.type) {
    case 'INCREMENT': return { ...state, count: state.count + 1 }
    case 'FETCH_START': return { ...state, loading: true, error: null }
    case 'FETCH_SUCCESS': return { ...state, loading: false, count: action.payload }
    case 'FETCH_ERROR': return { ...state, loading: false, error: action.error }
    default: return state
  }
}

function Counter() {
  const [state, dispatch] = useReducer(reducer, initialState)

  async function fetchCount() {
    dispatch({ type: 'FETCH_START' })
    try {
      const count = await api.getCount()
      dispatch({ type: 'FETCH_SUCCESS', payload: count })
    } catch (e) {
      dispatch({ type: 'FETCH_ERROR', error: e.message })
    }
  }

  return (
    <div>
      {state.loading && <Spinner />}
      {state.error && <Error message={state.error} />}
      <p>Count: {state.count}</p>
      <button onClick={() => dispatch({ type: 'INCREMENT' })}>+</button>

  )
}

## Custom Hooks — Extracting Stateful Logic

Custom hooks are the primary reuse mechanism in React. Any function that starts with `use` and calls built-in hooks is a custom hook.

```js
// useFetch — generic data fetching with loading/error states
function useFetch(url) {
  const [state, dispatch] = useReducer(
    (s, a) => ({ ...s, ...a }),
    { data: null, loading: true, error: null }
  )

  useEffect(() => {
    let cancelled = false
    dispatch({ loading: true, error: null })

    fetch(url)
      .then(r => r.json())
      .then(data => { if (!cancelled) dispatch({ data, loading: false }) })
      .catch(err => { if (!cancelled) dispatch({ error: err.message, loading: false }) })

    return () => { cancelled = true }
  }, [url])

  return state
}

// Usage
function UserProfile({ userId }) {
  const { data, loading, error } = useFetch(`/api/users/${userId}`)

  if (loading) return <Spinner />
  if (error) return <Error message={error} />
  return <Profile user={data} />
}
```

| Mistake | Fix |
|---------|-----|
| `setCount(count + 1)` in async handler | Use `setCount(c => c + 1)` |
| Missing dependency in `useEffect` | Add it or extract to a ref |
| Calling a hook inside a condition | Move condition inside the hook |

| Using `useLayoutEffect` when `useEffect` works | `useLayoutEffect` blocks paint — only for DOM measurements |

## The Rendering Mental Model

The App Router introduces a fundamental split: **Server Components** (RSC) and **Client Components**. Understanding where code runs is the foundation of everything else.

```
┌─────────────────────────────────────────────────────┐
│               Server Components (default)            │
│  ✓ Access DB/filesystem directly                    │
│  ✓ Zero JS shipped to browser                       │
│  ✓ Can import server-only libraries                 │
│  ✗ No useState, useEffect, event handlers           │
│  ✗ No browser APIs                                  │
├─────────────────────────────────────────────────────┤
│         Client Components ('use client')             │
│  ✓ useState, useEffect, useCallback                 │
│  ✓ Event handlers, animations                       │
│  ✓ Browser APIs (window, localStorage)              │
│  ✗ Cannot import server-only code                   │
└─────────────────────────────────────────────────────┘
```

The key insight: **push interactivity to the leaves**. Keep page/layout/data-fetching logic as RSC, add `'use client'` only at the interactive leaf components.

export function ThemeToggle() {
  const [dark, setDark] = useState(false)

  useEffect(() => {
    // Persist preference
    const saved = localStorage.getItem('theme')
    if (saved === 'dark' || (!saved && window.matchMedia('(prefers-color-scheme: dark)').matches)) {
      setDark(true)
      document.documentElement.classList.add('dark')
    }
  }, [])

  function toggle() {
    setDark(d => {
      const next = !d
      document.documentElement.classList.toggle('dark', next)
      localStorage.setItem('theme', next ? 'dark' : 'light')
      return next
    })
  }

  return (
    <button onClick={toggle} aria-label="Toggle theme">
      {dark ? '☀️' : '🌙'}
    </button>
  )
}

## What is O-RAN?

The O-RAN Alliance defines an open, interoperable radio access network architecture. Traditional RAN used proprietary, vertically integrated hardware from a single vendor. O-RAN disaggregates the base station into separate functional units connected by standardized open interfaces.

## Functional Split: CU / DU / RU

```
┌─────────────────────────────────────────┐
│              Core Network (5GC)          │
└──────────────────┬──────────────────────┘
                   │ N2/N3
┌──────────────────▼──────────────────────┐
│    CU — Centralized Unit                │
│  ┌─────────────┐  ┌─────────────────┐   │
│  │  CU-CP      │  │   CU-UP         │   │
│  │ (RRC, PDCP-C)│  │ (PDCP-U, SDAP) │   │
│  └─────────────┘  └─────────────────┘   │
└──────────────────┬──────────────────────┘
                   │ F1 interface
┌──────────────────▼──────────────────────┐
│    DU — Distributed Unit                │
│    (RLC, MAC, High-PHY)                 │
└──────────────────┬──────────────────────┘
                   │ Fronthaul (Open FH / eCPRI)
┌──────────────────▼──────────────────────┐
│    RU — Radio Unit                      │
│    (Low-PHY, RF)                        │
└─────────────────────────────────────────┘
```

## RIC: RAN Intelligent Controller

The RIC is O-RAN's AI/ML control plane — it monitors and optimizes the RAN in real time.

### Near-RT RIC (Near Real-Time)
- Control loop: 10ms – 1s
- Hosts **xApps** (third-party applications)
- Functions: interference management, load balancing, handover optimization
- Interface: **E2** to DU/CU

### Non-RT RIC
- Control loop: > 1s
- Hosts **rApps**
- Functions: policy management, AI/ML model training, long-term optimization
- Interface: **A1** to Near-RT RIC, **O1** to all O-RAN nodes

## Open Interfaces Summary

| Interface | Between | Protocol |
|-----------|---------|----------|
| O1 | SMO ↔ All nodes | NETCONF/YANG |
| A1 | Non-RT RIC ↔ Near-RT RIC | REST/JSON |
| E2 | Near-RT RIC ↔ CU/DU | ASN.1 |
| F1 | CU ↔ DU | GTP-U / SCTP |
| Open FH | DU ↔ RU | eCPRI over Ethernet |
| Xn | CU ↔ CU (inter-node) | SCTP |

## Fronthaul: eCPRI and CPRI

The fronthaul link between DU and RU is timing-critical. O-RAN uses **eCPRI** (enhanced CPRI) over standard Ethernet with IEEE 1588 (PTP) for synchronization.

Functional split option **7.2x** is the O-RAN standard split point:
- DU handles: High-PHY (FEC, scrambling, layer mapping)
- RU handles: Low-PHY (FFT/IFFT, beamforming, DAC/ADC)

## Benefits of Open RAN

1. **Vendor diversity** — mix RU from vendor A, DU from vendor B
2. **Cost reduction** — COTS hardware instead of proprietary ASICs
3. **Innovation** — third-party xApps can optimize the network
4. **Cloudification** — CU can run as a cloud-native VNF/CNF

## Challenges

- Fronthaul latency and jitter requirements are strict (< 100μs)
- Multi-vendor integration testing is complex
- Security surface increases with more open interfaces
- Performance gap vs. traditional integrated RAN still exists in some deployments

## What a Closure Actually Is

A closure is a function bundled with its **lexical environment** — the scope chain at the point where the function was *defined*, not where it's called.

```js
function outer() {
  const secret = 42  // Lives in outer's scope

  function inner() {
    console.log(secret)  // inner "closes over" secret
  }

  return inner  // Return the function, scope travels with it
}

const fn = outer()  // outer() has returned, its stack frame is gone
fn()  // 42 — but secret is still alive, held by the closure
```

The garbage collector cannot collect `secret` because `inner` holds a live reference to it through its closure. The scope lives as long as the function lives.

## Factory Functions — Closures for Encapsulation

The classic use case: create private state that only your returned functions can access.

```js
function makeCounter(start = 0) {
  let count = start  // Private — no external access

  return {
    increment() { count++ },
    decrement() { count-- },
    reset()     { count = start },
    value()     { return count },
  }
}

const counter = makeCounter(10)
counter.increment()
counter.increment()
console.log(counter.value())  // 12
console.log(counter.count)    // undefined — truly private
```

Each call to `makeCounter` creates a **new closure** with its own `count`:

```js
const a = makeCounter()
const b = makeCounter()
a.increment()
a.increment()
console.log(a.value())  // 2
console.log(b.value())  // 0 — independent state
```

## The Closure-in-Loop Gotcha

This is the most common closure bug in JavaScript. Every interviewer asks about it.

```js
// ❌ Classic bug
for (var i = 0; i < 3; i++) {
  setTimeout(() => console.log(i), 100)
}
// Prints: 3, 3, 3
// All three callbacks close over the *same* i (var is function-scoped)
// By the time they run, the loop has ended and i === 3
```

**Fix 1: Use `let` (block-scoped — creates new binding per iteration)**
```js
for (let i = 0; i < 3; i++) {
  setTimeout(() => console.log(i), 100)
}
// Prints: 0, 1, 2  ✓
```

**Fix 2: IIFE to capture the value**
```js
for (var i = 0; i < 3; i++) {
  ;(function(j) {
    setTimeout(() => console.log(j), 100)
  })(i)
}
// Prints: 0, 1, 2  ✓ — j is a new variable per IIFE call
```

**Fix 3: Extract to a factory function**
```js
function makeLogger(n) {
  return () => console.log(n)  // n is captured at call time
}

for (var i = 0; i < 3; i++) {
  setTimeout(makeLogger(i), 100)
}
// Prints: 0, 1, 2  ✓
```

## Closures in React Hooks

React hooks rely entirely on closures. Each render creates new closures, which is why the rules of hooks (no conditionals, no loops) exist — React tracks hooks by call order, and the closures must be stable.

```js
function useDebounce(value, delay) {
  const [debouncedValue, setDebouncedValue] = useState(value)

  useEffect(() => {
    // This closure captures 'value' at this render's snapshot
    const timer = setTimeout(() => {
      setDebouncedValue(value)  // Always the 'value' from *this* effect's render
    }, delay)

    return () => clearTimeout(timer)  // Cleanup closes over 'timer'
  }, [value, delay])

  return debouncedValue
}
```

## Memory Considerations

Closures can leak memory if they inadvertently hold large objects:

function processLargeData() {
  const bigArray = new Array(1_000_000).fill(0)  // 8 MB

  return () => console.log(bigArray.length)

function processLargeData() {
  const bigArray = new Array(1_000_000).fill(0)
  const length = bigArray.length  // Extract what's needed
  // bigArray can now be GC'd after the function returns

  return () => console.log(length)
}
```

## Rebase vs Merge

```
Merge (preserves history, adds merge commit):

  main:    A──B──C──────────M
                  \        /
  feature:         D──E──F

Result: "Merge feature/xyz into main" commit
Use for: merging release branches, preserving context

Rebase (linear history, replays commits):

  main:    A──B──C──D'──E'──F'
  feature: (gone — rebased onto main)

Result: Clean linear history, as if you always had the latest main
Use for: feature branches before merging to main
```

**The golden rule:** Never rebase shared branches (main, release). Only rebase your own local branches.

## Grid vs Flexbox — The Decision

```
One axis (row OR column)?      → Flexbox
Two axes (row AND column)?     → Grid
Unknown number of items, auto-wrap? → Grid with auto-fit
Centered single item?          → Either (grid: place-items: center)
```

They're complementary, not competing. Use grid for **layout**, flexbox for **component internals**.

## Defining the Grid

### Explicit Columns and Rows

```css
.container {
  display: grid;
  grid-template-columns: 200px 1fr 1fr;  /* Fixed | flexible | flexible */
  grid-template-rows: 100px auto;         /* Fixed height | content height */
  gap: 16px;                              /* gap between all cells */
  /* column-gap: 24px; row-gap: 8px;  — different values */

`fr` allocates the **remaining** space after fixed values are subtracted:

```css
/* 3 equal columns */

grid-template-columns: 280px 1fr;

/* 1:2:1 ratio */
grid-template-columns: 1fr 2fr 1fr;

## The repeat() Function

```css
/* 12 equal columns (common CSS grid system) */
grid-template-columns: repeat(12, 1fr);

/* 4 columns, each 200px */
grid-template-columns: repeat(4, 200px);

/* Pattern: 40px 200px 40px 200px ... (6 times = 12 tracks) */

## auto-fit vs auto-fill — Responsive Without Media Queries

Both with `minmax()` enable responsive grids without a single media query:

```css
/* auto-fit: collapses empty tracks, items stretch to fill */
grid-template-columns: repeat(auto-fit, minmax(250px, 1fr));

/* auto-fill: preserves empty track space */
grid-template-columns: repeat(auto-fill, minmax(250px, 1fr));
```

**Visual difference:**

```
Container width: 900px, min item width: 250px → 3 items fit

With 2 actual items:

auto-fit:   [  item 1  ][  item 2  ]  ← stretches to fill
auto-fill:  [  item 1  ][  item 2  ][        ]  ← preserves ghost track
```

**Rule of thumb:** Use `auto-fit` (items stretch to fill gaps) for almost everything. Use `auto-fill` when you need empty cells to take physical space (e.g., calendar grids).

Subgrid lets a child element participate in its parent's grid tracks. Browser support is now solid (all major browsers 2024+).

### 주요 네트워크 기능 (NF)

┌────────────────────────────────────────────────────────────────┐

│                    (HTTP/2 + JSON/Protobuf)                    │

### NR 뉴머롤로지 (Numerology)

5G NR은 서브캐리어 간격을 유연하게 설정할 수 있습니다.

| μ | SCS (kHz) | 슬롯 당 심볼 | 슬롯 길이 | 주요 용도 |
|---|-----------|------------|---------|---------|
| 0 | 15 | 14 | 1 ms | FR1, LTE 호환 |
| 1 | 30 | 14 | 0.5 ms | FR1 일반 NR |
| 2 | 60 | 14 | 0.25 ms | FR1/FR2 |
| 3 | 120 | 14 | 0.125 ms | FR2 (mmWave) |
| 4 | 240 | 14 | 0.0625 ms | FR2 참조 신호 |

```python
def slot_duration_us(mu):
    """NR 슬롯 길이 계산"""
    scs_kHz = 15 * (2 ** mu)        # 서브캐리어 간격
    slot_per_subframe = 2 ** mu      # 서브프레임당 슬롯 수
    slot_duration = 1000 / slot_per_subframe  # us 단위
    return scs_kHz, slot_duration

for mu in range(5):
    scs, dur = slot_duration_us(mu)
    print(f"μ={mu}: SCS={scs:4d} kHz, 슬롯={dur:.4f} ms")

# μ=0: SCS=  15 kHz, 슬롯=1.0000 ms
# μ=1: SCS=  30 kHz, 슬롯=0.5000 ms
# μ=2: SCS=  60 kHz, 슬롯=0.2500 ms
# μ=3: SCS= 120 kHz, 슬롯=0.1250 ms
# μ=4: SCS= 240 kHz, 슬롯=0.0625 ms
```

def hybrid_beamforming_weight(N_antenna, N_beam, angles_deg):
    """
    N_antenna: 안테나 수
    N_beam:    동시 빔 수
    angles_deg: 각 빔의 방향 (도)
    """
    d = 0.5  # 안테나 간격 (파장 기준)
    W = np.zeros((N_antenna, N_beam), dtype=complex)

    for i, theta in enumerate(angles_deg):
        theta_rad = np.deg2rad(theta)
        # 아날로그 스티어링 벡터
        steering = np.exp(1j * 2 * np.pi * d *
                          np.arange(N_antenna) * np.sin(theta_rad))
        W[:, i] = steering / np.sqrt(N_antenna)

    return W

# 8 안테나, 3개 빔 (-30°, 0°, +30°)
W = hybrid_beamforming_weight(8, 3, [-30, 0, 30])
print(f"빔포밍 가중치 행렬 크기: {W.shape}")  # (8, 3)
```

## 개요

NTN(Non-Terrestrial Network)은 위성, HAPS(High Altitude Platform Station), UAV 등 비지상 플랫폼을 활용하여 지상 네트워크를 보완하거나 대체하는 통신 인프라입니다. 5G NR Rel-17에서 NTN 표준화가 시작되었으며, 6G에서는 지상망과 비지상망의 완전한 통합(Unified Terrestrial and Non-Terrestrial Network)이 핵심 목표입니다.

## 주요 기술 과제

### 1. 대규모 전파 지연 (Large Propagation Delay)

LEO 기준 편도 최대 ~10 ms, GEO는 ~270 ms로 기존 지상망 대비 매우 큰 RTT를 가집니다.

**영향 및 대응 방안:**

```
문제: HARQ Stop-and-Wait → 처리량 급락
  └─ LEO(600km) 기준 RTT ≈ 4 ms × 2 = ~8 ms
  └─ GEO 기준 RTT ≈ 270 ms × 2 = ~540 ms

대응:
  ① HARQ 비활성화 (Rel-17 NTN: HARQ 재전송 횟수 확장)
  ② Extended HARQ Round-Trip Time 허용
  ③ ARQ 상위 계층(RLC AM)에서 재전송 담당
```

### 2. 도플러 편이 (Doppler Shift)

LEO 위성은 초속 약 7.5 km/s로 이동하며 최대 수십 kHz의 주파수 편이가 발생합니다.

```python
# 도플러 편이 계산 예시
def doppler_shift(freq_hz, velocity_ms, angle_deg):
    """
    freq_hz    : 반송파 주파수 (Hz)
    velocity_ms: 위성 속도 (m/s)
    angle_deg  : 위성 이동 방향과 시선 방향 각도
    """
    import math
    c = 3e8  # 광속 (m/s)
    theta = math.radians(angle_deg)
    delta_f = freq_hz * (velocity_ms / c) * math.cos(theta)
    return delta_f

# LEO 위성 (7500 m/s), Ka 대역 (20 GHz), 정면 통과 시
shift = doppler_shift(20e9, 7500, 0)
print(f"최대 도플러 편이: {shift/1e3:.1f} kHz")  # ≈ 500 kHz
```

**대응 방안:**
- UE 측 또는 네트워크 측 Pre-compensation
- 위성 궤도 정보 기반 예측 보정 (Ephemeris-based)
- Wide-guard-band 설계

### 3. 타이밍 어드밴스 (Timing Advance)

지상망 대비 수백 배 이상의 TA 값이 필요합니다.

| 시스템 | 최대 TA 범위 |
|--------|-------------|
| 지상 LTE/NR | ~0.67 ms (최대 ~100 km) |
| LEO NTN (Rel-17) | ~27 ms (고도 600 km) |
| GEO NTN | ~541 ms |

- 3GPP Rel-17: TA 범위 확장 (최대 66,460 Tc 단계 → NTN용 확장)
- UE 자체 GNSS 기반 사전 보정(Pre-compensation) 도입

## 1. LEO (Low Earth Orbit)

### 궤도 역학

```
케플러 제3법칙:  T² = (4π²/GM) × a³

고도 550 km 기준:
  반장축 a = 6,371 + 550 = 6,921 km
  공전 주기 T ≈ 95.5분
  궤도 속도 v = 2πa/T ≈ 7.59 km/s
```

### 자유공간 경로손실 (FSPL)

```python
import math

def fspl_dB(distance_km, freq_GHz):
    c = 3e8
    f = freq_GHz * 1e9
    d = distance_km * 1e3
    lam = c / f
    return 20 * math.log10(4 * math.pi * d / lam)

# Ka-band (26.5 GHz), 고도별 경로손실
for alt in [550, 1200, 2000]:
    loss = fspl_dB(alt, 26.5)
    print(f"고도 {alt:5d} km: {loss:.1f} dB")

# 출력:
# 고도   550 km: 170.3 dB
# 고도  1200 km: 176.0 dB
# 고도  2000 km: 179.5 dB
```

### LEO 도플러 특성

LEO 위성은 빠른 이동으로 인해 가장 큰 도플러 편이가 발생합니다.

```
최대 도플러 편이 (Ka-band 26.5 GHz, LEO 550 km, 수직 통과):

  Δf_max = f_c × (v / c) = 26.5×10⁹ × (7590 / 3×10⁸) ≈ 670 kHz

  ← 위성 접근 시: +670 kHz
  → 위성 이탈 시: -670 kHz
  수직 통과 시간: 약 10분 (仰角 10° 이상 기준)
```

**주파수 편이 보정 방법:**
- UE 사전 보정 (Pre-compensation): GNSS 위치 + 위성 궤도력(Ephemeris) 활용
- 네트워크 기반 보정: 지상국에서 공통 도플러 제거 후 링크 유지
- OFDM 서브캐리어 간격 확대: 도플러 대비 파일럿 설계

### LEO 핸드오버

```
LEO 위성 가시 시간: 약 5 ~ 15분 (仰角 10° 기준)

핸드오버 전략:

  [위성 A] ──── 가시 시간 ────►[위성 B]
      │                             │
      │   ← 핸드오버 결정 구간 →    │
      └──── 소프트 핸드오버 ────────┘

  - 위성간 핸드오버 빈도: ~6 회/시간
  - ISL(위성 간 링크) 활용 시 지상국 거치지 않고 연속성 유지
  - Make-before-break 방식 필수
```

## 2. MEO (Medium Earth Orbit)

### 궤도 역학

```
대표 고도: 10,000 ~ 20,200 km

GPS (20,200 km):
  공전 주기 ≈ 12시간
  궤도 속도 ≈ 3.87 km/s

O3b MEO (8,063 km):
  공전 주기 ≈ 4.8시간
  RTT ≈ 54 ms
```

### MEO 주요 특성

```
장점:
  ✓ LEO 대비 넓은 커버리지 (위성 수 적게 필요)
  ✓ GEO 대비 낮은 전파 지연
  ✓ 반알렌대(Van Allen Belt) 일부 회피 가능
  ✓ 지상 추적 장비 부담 완화

단점:
  ✗ 반알렌대 일부 구간 통과 → 방사선 환경 심각
  ✗ LEO 대비 여전히 핸드오버 필요
  ✗ GEO 대비 더 많은 위성 필요
```

### 방사선 환경 (Van Allen Belt)

고도별 방사선 강도:

    0         5,000      15,000      35,786 km

    │                 ████████
    │           ████████████████               ← 내부 Van Allen 대
    │     ██████████████████████████████
    │ ████████████████████████████████████████  ← 외부 Van Allen 대

    낮음   높음         낮음         없음

## 3. GEO (Geostationary Earth Orbit)

### 궤도 역학

```
GEO 조건: 공전 주기 = 지구 자전 주기 = 23시간 56분 4초

정지 궤도 고도 계산:
  T = 2π × √(a³/GM)
  a = ∛(GM × T²/4π²)
    = ∛(3.986×10¹⁴ × (86164)² / 4π²)
    ≈ 42,164 km (지구 중심 기준)
    = 42,164 - 6,371 ≈ 35,793 km (지표 기준)

적도 상공 고정 → 위도 ±81° 이내 커버
극지방 커버 불가 (仰角 < 5°)
```

### GEO 링크 버짓 예시

```python
def link_budget(eirp_dBW, fspl_dB, gt_dB_K, bandwidth_MHz):
    """
    EIRP   : 송신 등가등방복사전력 (dBW)
    FSPL   : 자유공간 경로손실 (dB)
    G/T    : 수신 시스템 성능 지수 (dB/K)
    BW     : 수신 대역폭 (MHz)
    """
    k = -228.6  # 볼츠만 상수 (dBW/K/Hz)
    bw_dBHz = 10 * math.log10(bandwidth_MHz * 1e6)

    cn0 = eirp_dBW - fspl_dB + gt_dB_K - k  # C/N0 (dBHz)
    cn  = cn0 - bw_dBHz                       # C/N  (dB)
    return cn0, cn

import math
cn0, cn = link_budget(
    eirp_dBW = 50,      # 위성 EIRP 50 dBW
    fspl_dB  = 209.6,   # GEO Ka-band 26.5 GHz
    gt_dB_K  = 18.0,    # 지상국 G/T 18 dB/K
    bandwidth_MHz = 36  # 36 MHz 트랜스폰더
)
print(f"C/N₀: {cn0:.1f} dBHz")
print(f"C/N : {cn:.1f} dB")
```

### GEO 전파 지연의 영향

```
GEO RTT ≈ 540 ms

TCP 성능 영향:
  - 윈도우 크기 W = 64 KB 기준:
    처리량 = W / RTT = 65,536 × 8 / 0.54 ≈ 970 kbps
    → 1 Mbps도 안 되는 TCP 성능 (물리 대역폭이 수백 Mbps여도)

  해결책:
  ① PEP (Performance Enhancing Proxy): TCP 분할
  ② 대용량 TCP 윈도우 (TCP SACK + 윈도우 스케일링)
  ③ QUIC / UDP 기반 프로토콜로 전환
  ④ DVB-S2X: 전용 위성 전송 표준 (ACM, VCM)
```

## RFSoC란?

Xilinx RFSoC(Radio Frequency System-on-Chip)는 고속 ADC/DAC와 FPGA 패브릭, ARM 프로세서를 하나의 칩에 통합한 플랫폼입니다. 기존의 분리형 RF 프론트엔드 + FPGA 구조 대비 다음의 장점을 제공합니다.

| 항목 | 기존 구조 | RFSoC |
|------|-----------|--------|
| ADC/DAC | 외부 칩 | 내장 (최대 16채널) |
| 샘플링 레이트 | ~500 MSPS | ~5 GSPS |
| 레이턴시 | 높음 | 매우 낮음 |
| 보드 면적 | 크다 | 컴팩트 |
| 소비 전력 | 높음 | 낮음 |

## 5G NR 적용

5G NR mmWave(FR2) 대역에서 RFSoC는 핵심 역할을 합니다.

### Massive MIMO Beamforming

```python
# 단순화된 디지털 빔포밍 예시
import numpy as np

def digital_beamforming(signals, steering_angle_deg, array_spacing=0.5):
    """
    ULA(Uniform Linear Array) 기반 디지털 빔포밍
    """
    N = len(signals)
    theta = np.deg2rad(steering_angle_deg)

    # 스티어링 벡터 계산
    d = array_spacing  # 파장 대비 간격
    steering_vec = np.exp(1j * 2 * np.pi * d * np.arange(N) * np.sin(theta))

    # 빔포밍 가중치 적용
    weighted = signals * steering_vec.conj()
    return np.sum(weighted)

# 8 안테나 기준, 30도 방향 빔 형성
signals = np.random.randn(8) + 1j * np.random.randn(8)
output = digital_beamforming(signals, steering_angle_deg=30)
```

## 개요

LTE(Long Term Evolution)의 무선 접속 계층(Radio Access Layer)은 여러 서브계층으로 구성됩니다. 이 문서는 각 계층의 역할과 상호작용을 정리합니다.

## 계층 구조

```
┌───────────────────────┐
│      Application      │
├───────────────────────┤
│   IP / Non-IP (NAS)   │
├───────────────────────┤
│         PDCP          │  ← 헤더 압축, 암호화
├───────────────────────┤
│          RLC          │  ← ARQ, 분할/재조합
├───────────────────────┤
│          MAC          │  ← 스케줄링, HARQ
├───────────────────────┤
│          PHY          │  ← OFDMA/SC-FDMA, 변조
└───────────────────────┘
```

## PHY 계층 (Physical Layer)

PHY 계층은 실제 무선 신호 송수신을 담당합니다.

- **다운링크**: OFDMA (Orthogonal Frequency Division Multiple Access)
- **업링크**: SC-FDMA (Single Carrier FDMA)
- **변조 방식**: QPSK, 16QAM, 64QAM
- **채널 코딩**: Turbo Code

### Resource Block 구조

| 항목 | 값 |
|------|-----|
| 서브캐리어 간격 | 15 kHz |
| RB당 서브캐리어 수 | 12 |
| 슬롯 당 심볼 수 | 7 (Normal CP) |
| TTI | 1ms (1 서브프레임) |

## MAC 계층 (Medium Access Control)

### HARQ (Hybrid ARQ)

HARQ는 오류 정정에 있어 ARQ와 FEC를 결합한 방식입니다.

```c
/* HARQ Process 상태 예시 (의사코드) */
typedef struct {
    uint8_t process_id;
    bool    ack_received;
    uint8_t retx_count;
    uint8_t rv;           /* Redundancy Version */
} harq_process_t;

void harq_feedback_handler(harq_process_t *proc, bool ack) {
    if (ack) {
        proc->ack_received = true;
        proc->retx_count = 0;
    } else {
        proc->retx_count++;
        proc->rv = (proc->rv + 1) % 4;  /* RV: 0, 2, 3, 1 순환 */
    }
}
```

## RLC 계층 (Radio Link Control)

RLC는 세 가지 동작 모드를 지원합니다.

- **TM (Transparent Mode)**: 처리 없이 투명 전달
- **UM (Unacknowledged Mode)**: 단방향, 재전송 없음
- **AM (Acknowledged Mode)**: 양방향, ARQ 재전송

## PDCP 계층 (Packet Data Convergence Protocol)

- 헤더 압축: ROHC (Robust Header Compression)
- 암호화: AES-128 (EEA2), Snow 3G (EEA1)
- 무결성 보호: HMAC-SHA-256 (EIA2)

## 정리

LTE 프로토콜 스택의 각 계층은 독립적인 역할을 수행하면서, 인터페이스를 통해 데이터를 교환합니다. 모뎀 SW 개발 시에는 이 계층 분리 원칙을 유지하는 것이 유지보수성에 중요합니다.

## 개요

군용 위성통신(Military Satellite Communication, MILSATCOM)은 상용 위성통신과 달리 **항재밍(Anti-Jamming)**, **저피탐(LPI/LPD)**, **고가용성** 요구사항이 핵심입니다.

## 주파수 대역별 특성

| 대역 | 주파수 | 특성 | 주요 용도 |
|------|--------|------|-----------|
| UHF | 300~3000 MHz | 기동성 우수, 소형 안테나 | 전술 단말, 이동체 |
| SHF (X) | 7.25~8.4 GHz | 균형적 성능 | 기본 전술 통신 |
| SHF (Ka) | 26.5~40 GHz | 고용량, 강우 영향 | 전략 통신, 브로드밴드 |
| EHF | 30~300 GHz | 항재밍 최강 | 핵심 전략 통신 |

## 링크 버짓 기본 구조

위성통신 링크 버짓은 신호가 지상 → 위성 → 지상을 거치면서의 손실/이득을 계산합니다.

```
EIRP(dBW) - FSL(dB) - 기타손실(dB) + G/T(dB/K) - k(dBW/K/Hz) = C/N₀(dBHz)

여기서:
  EIRP = 등가등방복사전력 (Tx Power + Antenna Gain)
  FSL  = 자유공간 경로손실 = 20·log(4πd/λ)
  G/T  = 수신 성능 지수 (안테나 이득 - 잡음 온도)
  k    = 볼츠만 상수 (-228.6 dBW/K/Hz)
```

### 예시 계산 (GEO 위성, X-band)

```python
import math

def free_space_loss_dB(distance_km, freq_GHz):
    """자유공간 경로손실 계산"""
    c = 3e8  # m/s
    f = freq_GHz * 1e9
    d = distance_km * 1e3
    lam = c / f
    fsl = 20 * math.log10(4 * math.pi * d / lam)
    return fsl

# GEO 위성 (약 36,000 km), X-band 8 GHz
fsl = free_space_loss_dB(36000, 8.0)
print(f"자유공간 경로손실: {fsl:.1f} dB")  # ≈ 205.6 dB
```
