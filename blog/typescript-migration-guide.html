Migrating React Projects to TypeScript

May 15, 2026·5 min read

Migrating a JavaScript React project to TypeScript is not about changing everything at once. The realistic strategy is to gradually convert files while maintaining stability. This guide covers patterns and pitfalls encountered during actual migration.

Why TypeScript?

The real value of TypeScript is not "type safety" but fast feedback. Runtime errors appear immediately in the editor, and during refactoring, the compiler tells you the scope of impact.

// JS: error discovered at runtime
function getUser(id) {
  return fetch(`/api/users/${id}`).then(r => r.json());
}
getUser(null); // API call made, then server errors → delayed discovery

// TS: error displayed immediately in editor
function getUser(id: string): Promise<User> {
  return fetch(`/api/users/${id}`).then(r => r.json());
}
getUser(null); // ❌ error caught at edit time

Migration Preparation

1. Install TypeScript

npm install --save-dev typescript @types/react @types/react-dom @types/node

2. Configure tsconfig.json

Start loose and gradually tighten during migration:

{
  "compilerOptions": {
    "target": "ES2020",
    "lib": ["dom", "dom.iterable", "esnext"],
    "allowJs": true,           // allow JS files (incremental migration)
    "skipLibCheck": true,
    "strict": false,           // false initially, true later
    "noImplicitAny": false,    // false initially
    "jsx": "react-jsx",
    "moduleResolution": "bundler",
    "baseUrl": ".",
    "paths": {
      "@/*": ["src/*"]
    }
  },
  "include": ["src"]
}

3. File Extension Conversion Strategy

Priority: files with fewer dependencies first
Utility functions (.js → .ts)
  → Type definition files (create new)
    → Hooks (.js → .ts)
      → Components (.jsx → .tsx)
        → Pages (.jsx → .tsx)

Step-by-Step Conversion

Step 1: Type Definition Files First

// src/types/index.ts
export interface User {
  id: string;
  name: string;
  email: string;
  role: 'admin' | 'user' | 'viewer';
  createdAt: Date;
}

export interface ApiResponse<T> {
  data: T;
  message: string;
  success: boolean;
}

export interface PaginatedResponse<T> extends ApiResponse<T[]> {
  total: number;
  page: number;
  limit: number;
}

Step 2: Convert Utility Functions

// src/utils/date.ts (converted from date.js)

// Before (JS)
export function formatDate(date, format) {
  // ...
}

// After (TS)
type DateFormat = 'YYYY-MM-DD' | 'MM/DD/YYYY' | 'DD.MM.YYYY';

export function formatDate(date: Date | string, format: DateFormat = 'YYYY-MM-DD'): string {
  const d = typeof date === 'string' ? new Date(date) : date;
  
  const year = d.getFullYear();
  const month = String(d.getMonth() + 1).padStart(2, '0');
  const day = String(d.getDate()).padStart(2, '0');
  
  return format
    .replace('YYYY', String(year))
    .replace('MM', month)
    .replace('DD', day);
}

Step 3: Convert React Components

// UserCard.tsx (converted from UserCard.jsx)

// Before (JSX)
export function UserCard({ user, onEdit, className }) {
  return (
    <div className={className}>
      <h2>{user.name}</h2>
      <button onClick={() => onEdit(user.id)}>Edit</button>
    </div>
  );
}

// After (TSX)
interface UserCardProps {
  user: User;
  onEdit: (userId: string) => void;
  className?: string;
}

export function UserCard({ user, onEdit, className }: UserCardProps) {
  return (
    <div className={className}>
      <h2>{user.name}</h2>
      <button onClick={() => onEdit(user.id)}>Edit</button>
    </div>
  );
}

Step 4: Convert Custom Hooks

// hooks/useUser.ts

interface UseUserReturn {
  user: User | null;
  loading: boolean;
  error: string | null;
  refetch: () => void;
}

export function useUser(userId: string): UseUserReturn {
  const [user, setUser] = useState<User | null>(null);
  const [loading, setLoading] = useState(true);
  const [error, setError] = useState<string | null>(null);

  const fetchUser = useCallback(async () => {
    try {
      setLoading(true);
      const response = await fetch(`/api/users/${userId}`);
      const data: ApiResponse<User> = await response.json();
      setUser(data.data);
    } catch (err) {
      setError(err instanceof Error ? err.message : 'Unknown error');
    } finally {
      setLoading(false);
    }
  }, [userId]);

  useEffect(() => { fetchUser(); }, [fetchUser]);

  return { user, loading, error, refetch: fetchUser };
}

Common Problems and Solutions

Problem 1: Overusing the any Type

// Bad pattern — any abandons type safety
function processData(data: any) {
  return data.value.toUpperCase(); // potential runtime error
}

// Good pattern — use unknown with type guard
function processData(data: unknown) {
  if (typeof data === 'object' && data !== null && 'value' in data) {
    const value = (data as { value: unknown }).value;
    if (typeof value === 'string') {
      return value.toUpperCase();
    }
  }
  throw new Error('Unexpected data format');
}

Problem 2: Missing Types for External Libraries

# Install @types packages
npm install --save-dev @types/lodash @types/moment

# For libraries without types, declare manually
# src/types/declarations.d.ts
declare module 'untyped-library' {
  export function doSomething(value: string): number;
}

Problem 3: API Response Types

// Runtime validation + types together — using zod
import { z } from 'zod';

const UserSchema = z.object({
  id: z.string(),
  name: z.string(),
  email: z.string().email(),
  role: z.enum(['admin', 'user', 'viewer'])
});

type User = z.infer<typeof UserSchema>; // type auto-inferred

async function fetchUser(id: string): Promise<User> {
  const res = await fetch(`/api/users/${id}`);
  const raw = await res.json();
  return UserSchema.parse(raw); // validated at runtime too
}

Problem 4: Event Handler Types

// React event types
const handleChange = (e: React.ChangeEvent<HTMLInputElement>) => {
  setValue(e.target.value);
};

const handleSubmit = (e: React.FormEvent<HTMLFormElement>) => {
  e.preventDefault();
  // ...
};

const handleKeyDown = (e: React.KeyboardEvent<HTMLInputElement>) => {
  if (e.key === 'Enter') handleSubmit();
};

Enabling strict Mode

Gradually tighten tsconfig after basic conversion is complete:

{
  "compilerOptions": {
    "strict": true,              // enable all strict options
    "noUncheckedIndexedAccess": true,  // include undefined in array access
    "exactOptionalPropertyTypes": true, // strict optional property handling
    "noImplicitReturns": true,   // force return on all paths
    "noFallthroughCasesInSwitch": true
  }
}

Migration Completion Checklist

  • [ ] Change allowJs: false (all files are .ts/.tsx)
  • [ ] Enable strict: true and achieve 0 errors
  • [ ] 0 uses of any type (grep -r ": any")
  • [ ] Type definitions for all API responses
  • [ ] interface definitions for all component props
  • [ ] Add type tests (tsd or expect-type)

The real goal of TypeScript migration is not perfect type coverage. It's about shifting errors from runtime to compile time and maximizing the use of IDE autocompletion and refactoring tools.
