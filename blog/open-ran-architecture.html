Open RAN Architecture Overview

May 15, 2026·5 min read

Open RAN (Open Radio Access Network) is a technology paradigm that separates the hardware and software of mobile base stations and ensures interoperability through open interfaces. It opens up the traditionally proprietary base station equipment market, allowing components from different vendors to be combined.

Traditional RAN vs Open RAN

Traditional RAN (Closed RAN)

[Antenna] ─ [RRU] ─ [BBU] ─ [Core Network]
  ↑            ↑       ↑
Proprietary hardware supplied as integrated solution
by a single vendor (Ericsson, Nokia, Huawei, Samsung, etc.)

Single-vendor solutions offer high performance and stability, but suffer from vendor lock-in. Replacing equipment requires replacing the entire system.

Open RAN

[Antenna] ─ [O-RU] ─ [O-DU] ─ [O-CU] ─ [Core Network]
           Vendor A   Vendor B   Vendor C
              ↕          ↕          ↕
      Open interfaces (O-RAN Alliance standard)

Each component can be sourced from different vendors. Software runs on commodity servers (COTS).

O-RAN Alliance Architecture

Key Components

O-RU (O-RAN Radio Unit)

  • Handles antenna and RF processing
  • Radio signal transmission/reception, A/D and D/A conversion
  • Fronthaul interface: eCPRI (enhanced Common Public Radio Interface)
  • Processing: lower Physical Layer functions

O-DU (O-RAN Distributed Unit)

  • Real-time baseband processing
  • Upper Physical Layer, Layer 2 (MAC, RLC)
  • Latency-sensitive processing: response within a few ms required
  • Midhaul: connected to O-CU via F1 interface

O-CU (O-RAN Central Unit)

  • Non-real-time protocol processing
  • Layer 3 (PDCP), RRC (Radio Resource Control)
  • O-CU-CP: Control Plane
  • O-CU-UP: User Plane

RIC (RAN Intelligent Controller) Core innovation of O-RAN. Handles AI/ML-based wireless network optimization:

Near-RT RIC (10ms ~ 1s period)
├── xApp (third-party apps)
│   ├── Handover optimization app
│   ├── Interference management app
│   └── Traffic steering app
└── E2 interface → O-DU, O-CU

Non-RT RIC (>1s period)
├── rApp (non-real-time apps)
│   ├── Long-term trend analysis
│   └── ML model training/deployment
└── A1 interface → Near-RT RIC

Open Interfaces

| Interface | Connection | Key Characteristics | |-----------|-----------|---------------------| | Fronthaul (eCPRI) | O-RU ↔ O-DU | Low latency, high bandwidth, IEEE 802.1CM | | Midhaul (F1) | O-DU ↔ O-CU | 3GPP F1 interface | | Backhaul (N2/N3) | O-CU ↔ 5G Core | 3GPP NG interface | | E2 | O-DU/O-CU ↔ Near-RT RIC | O-RAN Alliance defined | | A1 | Non-RT RIC ↔ Near-RT RIC | Policy/ML model delivery | | O1 | All components ↔ SMO | Management, operations, and automation |

Cloud-Native Deployment

Open RAN components are packaged as containers and run on Kubernetes:

# O-DU deployment example (Kubernetes)
apiVersion: apps/v1
kind: Deployment
metadata:
  name: o-du
  namespace: oran
spec:
  replicas: 2
  template:
    spec:
      containers:
      - name: o-du
        image: oran/o-du:v2.1
        resources:
          limits:
            cpu: "16"
            memory: "32Gi"
            hugepages-1Gi: "8Gi"  # for real-time processing performance
          requests:
            cpu: "12"
        securityContext:
          privileged: true  # kernel-level access required

xApp Development Pattern

xApps running in the Near-RT RIC collect and control wireless network data via the E2 interface:

# Basic xApp structure (Python SDK)
from ricxappframe.xapp_frame import RMRXapp

class HandoverOptimizationXApp:
    def __init__(self):
        self.xapp = RMRXapp(
            default_handler=self.handle_message,
            config_handler=self.handle_config
        )
    
    def handle_message(self, summary, sbuf):
        # Parse E2 message
        mtype = summary[rmr.RMR_MS_MSG_TP_SZ]
        
        if mtype == E2_INDICATION:
            # Collect KPI data: RSRP, SINR, PRB utilization
            kpi = self.parse_e2_indication(sbuf)
            
            # Handover decision via AI/ML model
            decision = self.ml_model.predict(kpi)
            
            if decision.should_handover:
                self.send_handover_control(decision.target_cell)
    
    def run(self):
        self.xapp.run()

Open RAN Challenges

1. Latency and Synchronization

The fronthaul between O-RU and O-DU requires latency in the tens to hundreds of microseconds range. IEEE 1588v2 (PTP) precision time synchronization is mandatory.

Requirements (Category A Split, 3GPP TR 38.801)
- Latency: &lt; 100μs (one-way)
- Jitter: &lt; ±65ns (IEEE 1588v2 Class C)
- Bandwidth: 100MHz channel × 4 MIMO = ~25Gbps

2. Multi-Vendor Integration Testing

While theoretically interoperable, subtle implementation differences between vendors cause compatibility issues in real multi-vendor environments. The O-RAN Alliance's OTIC (Open Testing and Integration Centre) operates certification programs.

3. Security

Open interfaces expand the attack surface:

  • xApp supply chain security (blocking malicious xApps)
  • E2 interface encryption (TLS 1.3 required)
  • O-Cloud container runtime security

O-RAN and 5G SA Relationship

O-RAN is the openization of the RAN (Radio Access Network) layer and independently combines with 5G Stand-Alone (SA) Core:

5G SA Core (open source options: OpenAirInterface, free5GC)
    ↓ N2/N3
O-CU (Open RAN)
    ↓ F1
O-DU (Open RAN)
    ↓ eCPRI
O-RU (Open RAN)
    ↓
Antenna

Fully open-source 5G base stations are possible (OpenAirInterface + srsRAN). Used for small-scale research deployments.

Summary

Open RAN transforms mobile base stations into a software-defined approach and enables a multi-vendor ecosystem. AI/ML-based dynamic optimization through the RIC is its core value, while cloud-native deployment realizes TCO reduction and operations automation. However, latency requirements, multi-vendor integration complexity, and security issues remain challenges to solve.
