Git Stash — A Complete Guide to Temporarily Saving Your Work

May 15, 2026·6 min read

Have you ever been deep in the middle of coding a feature, only to get an urgent request to fix a production bug? Or realized you've been working on the wrong branch? Git Stash is the tool that saves you in these situations.

In this guide, we'll cover everything you need to know about git stash — from basic commands to advanced workflows and real-world scenarios.

What Is Git Stash?

Git Stash is a built-in Git feature that temporarily saves uncommitted changes (both staged and unstaged) without creating a commit. Think of it as a clipboard for your working directory — you can stash your changes, do something else, and come back to restore them later.

When Should You Use It?

| Scenario | Why Stash Helps | |----------|-----------------| | Emergency bug fix mid-feature | Switch branches without losing progress | | Working on the wrong branch | Move changes to the correct branch | | Pulling remote updates with local changes | Avoid merge conflicts during git pull | | Applying the same fix across multiple branches | Reuse stashed changes on different branches |

Essential Git Stash Commands

1. Saving Changes

# Basic stash (no message)
# git stash

# Recommended: stash with a descriptive message
# git stash save "WIP: user authentication form validation"

# Include untracked (new) files
# git stash -u

# Stash specific files only
# git stash push -m "fix: header alignment" src/header.css

Always use descriptive messages — your future self will thank you when looking at a list of stashes.

2. Listing Stashes

# git stash list

Output example:

stash@{0}: On feature/login: WIP: user authentication form validation
stash@{1}: On main: fix: temporary navbar workaround
stash@{2}: On develop: WIP: API integration draft

Each stash is indexed starting from {0} (most recent).

3. Viewing Stash Contents

# Summary of changes
# git stash show

# Full diff (detailed view)
# git stash show -p

# View a specific stash
# git stash show -p stash@{2}

4. Restoring Changes

There are two ways to restore stashed changes:

# Pop: restore AND delete the stash
# git stash pop

# Apply: restore but KEEP the stash
# git stash apply

# Restore a specific stash
# git stash pop stash@{1}
# git stash apply stash@{2}

Pro tip: Use apply first for safety. If everything looks good, manually drop the stash afterward.

5. Deleting Stashes

# Delete a specific stash
# git stash drop stash@{0}

# Delete ALL stashes (careful!)
# git stash clear

Advanced Features

Create a Branch from a Stash

If your stashed changes have diverged too much from the current branch, you can create a new branch directly from a stash:

# git stash branch new-feature-branch stash@{0}

This creates the branch, checks it out, applies the stash, and drops it — all in one command.

Stash Only Staged Changes

# git stash --keep-index

This stashes everything but keeps your staged (indexed) changes in the working directory.

Real-World Workflows

Scenario 1: Emergency Bug Fix

You're working on a new feature when a critical bug is reported:

# Save current work
# git stash save "WIP: shopping cart feature"

# Switch to main and fix the bug
# git checkout main
# git checkout -b hotfix/payment-error

# ... fix the bug, commit, merge ...

# Return to your feature branch
# git checkout feature/shopping-cart
# git stash pop

Scenario 2: Wrong Branch

You accidentally started coding on main instead of your feature branch:

# Stash the accidental changes
# git stash save "Oops: started on wrong branch"

# Switch to the correct branch
# git checkout feature/user-profile

# Restore the changes here
# git stash pop

Scenario 3: Clean Pull

You have local changes but need to pull the latest remote updates:

# Stash local changes
# git stash save "WIP: before pull"

# Pull latest changes
# git pull origin main

# Restore your work
# git stash pop

# Resolve any conflicts if needed

Scenario 4: Apply Fix Across Multiple Branches

You made a fix that needs to go to both develop and staging:

# Stash the fix
# git stash save "fix: API timeout config"

# Apply to develop (keep stash)
# git checkout develop
# git stash apply
# git add . && git commit -m "fix: API timeout config"

# Apply to staging (pop to clean up)
# git checkout staging
# git stash pop
# git add . && git commit -m "fix: API timeout config"

Common Mistakes and How to Avoid Them

1. Forgotten Stashes

Stashes can pile up and be forgotten. Regularly check your stash list:

# git stash list

2. Accidental pop with Conflicts

If git stash pop causes conflicts, the stash is not deleted. Resolve the conflicts, then manually drop it:

# After resolving conflicts:
# git add .
# git stash drop

3. "Missing" Files After Stash

New (untracked) files won't be stashed by default. Use the -u flag:

# git stash -u

4. Stash Accumulation

Too many stashes become unmanageable. Follow these best practices:

  • Use meaningful messages for every stash
  • Pop or drop stashes as soon as you're done with them
  • Run git stash list periodically to audit
  • Use git stash clear when you're sure nothing is needed

Quick Reference

| Command | Description | |---------|-------------| | git stash save "msg" | Save with message | | git stash -u | Include untracked files | | git stash list | Show all stashes | | git stash show -p | View stash diff | | git stash pop | Restore and delete | | git stash apply | Restore and keep | | git stash drop stash@&#123;n} | Delete specific stash | | git stash clear | Delete all stashes | | git stash branch <name> | Create branch from stash | | git stash push -m "msg" <file> | Stash specific file |

Conclusion

Git Stash is one of those tools that seems minor until you need it — and then it becomes a lifesaver. Whether you're juggling multiple tasks, fixing urgent bugs, or just cleaning up before a pull, stash keeps your workflow smooth and your changes safe.

Key takeaways:

  • Always use descriptive messages with git stash save "msg"
  • Prefer apply over pop for safety
  • Include untracked files with -u when needed
  • Audit and clean your stash list regularly
  • Use git stash branch when changes have diverged significantly
