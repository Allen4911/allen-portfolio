CLAUDE.md & Hooks: Configuring Claude Code Behavior

May 15, 2026·4 min read

Claude Code is not just an AI coding assistant. Through the CLAUDE.md file and Hooks system, you can finely control AI behavior per project. This guide covers configuration patterns immediately usable in practice.

What is CLAUDE.md?

CLAUDE.md is a markdown file that Claude Code automatically reads when opening a project. Instructions written here are automatically applied to all conversations.

File Location Priority

~/.claude/CLAUDE.md          # Global settings (all projects)
/project-root/CLAUDE.md      # Project settings
/project-root/src/CLAUDE.md  # Subdirectory settings

Subdirectory files don't override parent files — they merge. The effective pattern is to put team rules in global settings and codebase-specific details in project settings.

Practical CLAUDE.md Writing

# Project Rules

## Tech Stack
- Next.js 14 App Router (no pages/ directory)
- TypeScript strict mode
- Tailwind CSS (no inline style attributes)

## Code Conventions
- Components: PascalCase, matching filename
- Hooks: must have use prefix
- Tests: write failing test first before implementation (TDD)

## Forbidden Actions
- No console.log commits
- No any types
- No commits with TODO comments

Splitting Files with @references

When CLAUDE.md grows long, split it with @filename syntax:

# CLAUDE.md
@.claude/tdd-rules.md
@.claude/api-conventions.md
@.claude/deployment-guide.md

Each file is managed independently, enabling team-based ownership separation.

Hooks System

Hooks are a mechanism that automatically executes shell commands on specific Claude Code events. Defined in settings.json.

Configuration Locations

~/.claude/settings.json       # global hooks
.claude/settings.json         # project hooks
.claude/settings.local.json   # local only (gitignore recommended)

Hook Event Types

| Event | Trigger Point | |-------|--------------| | PreToolUse | Immediately before tool execution | | PostToolUse | Immediately after tool execution | | Notification | When notification occurs | | Stop | When Claude response completes | | SubagentStop | When subagent completes |

Practical Hook Examples

Auto lint on code modification:

{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Edit|Write",
        "hooks": [
          {
            "type": "command",
            "command": "npm run lint --fix 2>&1 | tail -5"
          }
        ]
      }
    ]
  }
}

Force test execution before git push:

{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Bash",
        "hooks": [
          {
            "type": "command",
            "command": "echo $CLAUDE_TOOL_INPUT | grep -q 'git push' && npm test || true"
          }
        ]
      }
    ]
  }
}

Telegram notification on Claude response complete:

{
  "hooks": {
    "Stop": [
      {
        "hooks": [
          {
            "type": "command",
            "command": "curl -s -X POST 'https://api.telegram.org/bot$BOT_TOKEN/sendMessage' -d 'chat_id=$CHAT_ID&text=Claude+done'"
          }
        ]
      }
    ]
  }
}

Environment Variables and Security

Manage sensitive values in hooks with environment variables:

# .env.local (gitignore required)
CLAUDE_HOOK_SECRET=your-secret
BOT_TOKEN=telegram-bot-token

settings.local.json should not be included in git. Keep team-shared settings in settings.json and personal tokens in settings.local.json.

Practical Usage Patterns

1. Token Optimization (RTK integration)

{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Bash",
        "hooks": [
          {
            "type": "command",
            "command": "~/.local/bin/rtk intercept"
          }
        ]
      }
    ]
  }
}

2. File Change Tracking

{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Write|Edit",
        "hooks": [
          {
            "type": "command",
            "command": "echo \"[$(date)] Modified: $CLAUDE_TOOL_INPUT\" >> .claude/change-log.txt"
          }
        ]
      }
    ]
  }
}

3. Production File Protection

{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Write",
        "hooks": [
          {
            "type": "command",
            "command": "echo $CLAUDE_TOOL_INPUT | grep -q 'production' && echo 'BLOCKED: production file' && exit 1 || true"
          }
        ]
      }
    ]
  }
}

Permission Settings

Use the permissions section of settings.json to limit commands Claude can execute:

{
  "permissions": {
    "allow": [
      "Bash(npm run *)",
      "Bash(git status)",
      "Bash(git diff *)"
    ],
    "deny": [
      "Bash(rm -rf *)",
      "Bash(git push --force)"
    ]
  }
}

Commands not in allow request user approval before execution. In CI environments, the --dangerously-skip-permissions flag can bypass this, but use it carefully.

Summary

| Feature | Use Case | |---------|---------| | CLAUDE.md | Coding rules, prohibited patterns, tech stack | | @references | Modularizing large rule sets | | PreToolUse Hook | Blocking dangerous operations, pre-validation | | PostToolUse Hook | Auto lint/format, change tracking | | Stop Hook | Completion notifications, report generation | | permissions | Limiting tool execution scope |

Combining CLAUDE.md and Hooks can enforce consistent AI collaboration rules across an entire team, and clearly separating personal settings (settings.local.json) from team settings (settings.json) improves maintainability.
